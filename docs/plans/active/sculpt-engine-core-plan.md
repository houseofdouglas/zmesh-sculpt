## Decisions & Notes
- Task 01: pinned `vite@7` + `@vitejs/plugin-react@4.7.0` (not latest `vite@8`/`plugin-react@6`) — vitest@3.2's peer range caps at vite ^7, and plugin-react@6 requires vite ^8. Revisit when vitest publishes a vite-8-compatible major.
- Task 01: added `vitest.test.passWithNoTests: true` so `npm run test` exits 0 on an empty suite (Vitest's default is exit 1 with zero test files).
- Task 01: React 19 needs `import type { JSX } from 'react'` for component return types — the global `JSX` namespace isn't ambient the way older React types provided.
- Task 02: `noUncheckedIndexedAccess` makes `typedArray[i] += x` a type error (the read side is `number | undefined`). Hot-loop accumulation uses documented non-null assertions (`arr[i]!`) instead of per-element branches, since the offsets are provably in-bounds. Test fixture is a regular tetrahedron centered at the origin — by symmetry each vertex's normal exactly equals its normalized position, giving a precise (not just sign-check) outward-normal assertion.
- Task 03 — **important correctness finding**: a naive UV-sphere (duplicated seam column + duplicated pole vertices per column, the standard rendering-oriented construction) is only *positionally* closed, not *index*-watertight — poles are represented by many coincident-position-but-different-index vertices. For a renderer that's invisible, but for us a brush stroke or remesh touching a pole would move only one of those duplicates and tear the mesh open. Redesigned `buildSurfaceOfRevolution` (shared by sphere/egg/capsule) so poles are a single welded vertex with a triangle fan, and non-pole rings wrap via modulo (no seam duplication needed since there's no texturing in v1). Verified via a from-scratch directed-edge-pairing watertight+orientable test — this is the check that would have caught the naive version's bug. `block` uses 8 shared corner vertices for the same reason (not 24 per-face vertices), trading perfectly flat corner shading for guaranteed connectivity.
- Task 03: `egg`'s width is an approximation by design (smoothstep-blended taper has no closed-form peak radius); height, and both of `capsule`'s dimensions, are exact. Documented in code and reflected in the tests' tolerances.
- Task 04: `checkManifold` classifies 3 defect kinds — boundary-edge (1 occurrence, a hole), inconsistent-winding (2 occurrences, same direction — one face flipped), non-manifold-edge (3+ occurrences). Uses numeric edge keys (`u*vertexCount+v`) rather than string concatenation since this runs on every remesh output. This formalizes (with defect reporting) the informal directed-edge check written ad hoc for Task 03's tests — Task 03 can migrate to this validator later if useful, though its own local check remains valid.
- Task 11: added `clampBlendSigned` ([-1,1], for brushes whose step is a *fraction* of an existing offset) and `clampVectorMagnitude` (preserves direction, for brushes whose displacement isn't along one known axis) to `brush-kernel.ts`, alongside Draw/Inflate's scalar `clampDisplacement`. Pinch/Crease/Flatten all use signed `strength*falloff` directly as their blend, so invert falls out of the same formula with no per-brush sign special-casing: Pinch's `position - tangentOffset*blend` contracts toward the axis for positive blend and expands for negative (spread); Crease combines Pinch's tangential term with an along-normal term forced inward by default, so inverting flips both together into "raised ridge"; Flatten's `-signedDistance*blend` snaps onto the plane at blend=1 and pulls further away for negative blend (emboss).
- Task 11: **Flatten is the one genuinely two-pass brush** — the plane (centroid + averaged normal) is one quantity shared by the whole stamp, so it must be aggregated from every affected vertex's pre-stamp position/normal before any of them move. This still needs no scratch array (the shared quantity is 6 scalars, not a per-vertex buffer), so it stays allocation-free like the others.
- Task 11: registry (`brushes/index.ts`) covers all 6 stamp kernels via `StampBrushType`/`STAMP_BRUSH_KERNELS`; Grab is deliberately excluded (stroke-stateful, not a stamp kernel — lands in Task 12's engine-layer stroke lifecycle instead).
- Task 10: promoted `MAX_STAMP_DISPLACEMENT_MM`/`clampDisplacement` from Draw-local into `brush-kernel.ts` (shared by every displacement brush now that a second one needs it) and refactored Draw to use it — a small, low-risk cleanup as the template pattern solidifies.
- Task 10: **Smooth is deliberately Gauss-Seidel-style, not Jacobi** — it reads/writes `positions` in place while iterating `affectedIndices`, so a vertex processed later in the same stamp can see an already-smoothed neighbor from earlier in that same stamp. This is a legitimate, standard variant of Laplacian smoothing (not a bug) and is what keeps the hot loop allocation-free per the acceptance criteria; a Jacobi (snapshot-then-apply) variant would need a per-stamp scratch buffer. Verified via a distance-to-neighbor-average curvature proxy on an artificial spike (pushed one sphere vertex far outward), and confirmed Smooth-invert is a no-op (per the spec's Resolved Questions on Smooth invert).
- Task 10: Inflate verified via a signed-volume proxy (divergence-theorem sum over triangles, exact for any watertight mesh) — inflating the whole sphere increases volume, negative strength decreases it.
- Task 08: established the shared `BrushKernelContext`/`Stamp`/`BrushKernel` types (`src/core/brushes/brush-kernel.ts`) every later brush (Tasks 10-12) follows. Draw uses `stamp.normal` (one shared direction for the whole stamp), not each vertex's own `context.normals` entry — that per-vertex-normal displacement is Inflate's job (Task 10), not Draw's.
- Task 08 — **real correctness bug caught and fixed while writing the "affected-region normal-recompute" helper** (FR-4). First draft zeroed-and-resummed normals for every vertex touching any triangle in the found set, which silently under-counted "two-hop" vertices (ones sharing a triangle with a 1-ring neighbor but not themselves in the 1-ring region) — their normal would be computed from only a partial subset of their true incident triangles. Fixed by separating "which triangles to sum from" (`triangleIndices`) from "which vertices to actually update" (`vertexIndicesToUpdate`, explicit) in `recomputeNormalsForTriangles` (`src/core/mesh/normals.ts`) — a vertex outside the update set is now left completely alone rather than partially/incorrectly touched. Verified by comparing the recomputed region against a full mesh rebuild (must match) and confirming a vertex outside the region is bit-identical to its pre-recompute value. Deliberately scoped to the 1-ring region only (not chased further to 2-hop) per FR-4's own wording; `findTrianglesTouchingVertices` is an honest O(triangleCount)-per-call scan (no new precomputed vertex→triangle structure), flagged in-code as a candidate optimization if profiling ever shows it's a hot spot.
- Task 07: `buildSpatialHash` derives its default cell size from mesh scale/density (bounding-box volume / vertexCount, cube-rooted, ×2 slack) rather than a fixed brush radius — brush size changes continuously via a slider, so tuning cells to one radius would mean rebuilding on every tick. `updateVertexPosition` keeps the broad-phase buckets accurate as vertices move mid-stroke, without a full rebuild; the narrow-phase distance check always uses live positions passed in fresh, so results stay exact even between refreshes as long as a vertex hasn't left the searched cell range. Verified exact-match against brute force (several point/radius cases plus a moved-vertex case) and sublinear performance on a ~100k-vertex sphere (loose 2x margin to avoid CI flakiness).
- Task 06: `buildVertexAdjacency` returns CSR arrays (`offsets`/`neighbors`), built via a one-time Set-per-vertex dedup pass (acceptable since it runs only on topology change, not per-stroke) so the query side — what Task 10's Smooth brush will actually use — is plain array indexing with zero per-query allocation. Verified against a hand-derived regular octahedron (degree exactly 4 per vertex, opposite-vertex-excluded) rather than a generic fixture, so the expected neighbor sets could be checked exactly, not just structurally.
- Task 05 — **the plan's assumed API was wrong; corrected during the spike.** `Manifold.levelSet` (SDF→isosurface) is for constructive/procedural modeling, not resampling an existing mesh — using it would have meant hand-rolling our own SDF sampler. The actual fit, found by reading manifold-3d's real type definitions: `Manifold.ofMesh(mesh)` + `.refineToLength()`/`.simplify()` + `.getMesh()`, which operate directly on existing topology with the same manifold guarantee. Conversion is a direct pass-through (no merge vectors needed — our SculptMesh vertices are already welded). **Task 16 must build on this corrected API, not levelSet.** Full findings (measurements, the `Manifold`-needs-`.delete()`-but-`Mesh`-doesn't gotcha, and the accepted `sharp`-via-transitive-dependency audit finding) are in `docs/adr/2026-07-19-voxel-remesh-library.md`. Verdict: GO on manifold-3d.
- **Q-01 RESOLVED (Task 09, 2026-07-23): Max = 500,000 triangles.** ≥60fps (Med, NFR-01) holds to ~199k triangles; ≥30fps (Max) holds to ~499k. Measured via `npm run bench:sculpt`; full writeup in `docs/design/q01-triangle-budget-findings.md`. **Important connected finding**: per-stamp time scales almost exactly linearly with *total* mesh size (2.5x triangles → 2.5x time, consistently across all 6 sizes tested), not with the constant-size affected region a good spatial hash implies — confirms Task 08's `findTrianglesTouchingVertices` (an honest, documented O(triangleCount) scan) is the actual bottleneck at scale, exactly as its own doc comment predicted. Not a v1 blocker — the measured ceiling already comfortably covers the spec's provisional Low/Med/High/Max targets — but flagged for Task 16 (or a Task 08 revisit) as the place to look first if a materially higher Max is wanted later.
- manifold-3d de-risked early in Task 05 before committing to it in Task 16 (revisit ADR if the spike fails).
- Brush work split: Draw (08, template) → Smooth+Inflate (10) → Pinch+Crease+Flatten+registry (11); Grab is stroke-stateful (12).
- Smooth-invert is a no-op in v1 (sharpen deferred).
- Task 12: `StrokeSampler` (the six stamp brushes) is a distance-accumulator, not a per-pointer-event stamper — it walks each new segment (`lastSamplePoint` → new hit) and emits a stamp every `radius * STAMP_SPACING_FRACTION` (0.25) mm of travel, carrying over leftover sub-spacing distance across calls. This is what makes fast vs. slow strokes over the same path produce identical stamp coverage (FR-7): both are driven by path length, not call count or elapsed time. Stamp normals between two sparse samples are lerp-then-renormalize of the segment's two endpoint normals (falls back to the newer endpoint's normal if the blend is near-zero-length, i.e. two near-opposite normals).
- Task 12: an off-mesh `update(null)` sets a `gapPending` flag rather than immediately re-anchoring — this is deliberate: it defers the "where does the path resume" decision to the *next* real hit, so the stroke never interpolates a line of stamps through the gap (which could cut across empty space or the far side of the mesh, per the spec's edge-case table). Re-entering the surface after a gap behaves like a fresh touch-down (re-anchor + stamp immediately at the re-entry point), same as `begin`.
- Task 12: **Grab is deliberately not a `BrushKernel`** — every stamp kernel recomputes its affected set fresh from live positions each stamp, but Grab must rigidly carry the *same* vertex set (and per-vertex falloff weight) captured at touch-down for the whole drag; recomputing mid-drag would let the grabbed region slide off the vertices it originally grabbed as they move away from the original stamp center. `GrabStroke` therefore lives in `engine/stroke.ts` (constructed once per grab from the touch-down stamp + a spatial-hash query the caller ran) rather than `core/brushes/`. It reuses `computeFalloff` and the shared `clampVectorMagnitude` safety-cap helper from Task 08/11's `brush-kernel.ts` rather than duplicating them.
- Task 12: Grab's `update` derives the drag delta from consecutive hit points when the viewport doesn't supply `worldDelta` (spec marks `worldDelta` optional on `SurfaceHit`), so it works either way upstream chooses to report cursor motion.
- Task 12: per the spec's brush table, Grab's invert is "n/a (direction is the drag)" — implemented by taking `Math.abs(stamp.strength)` as the per-vertex pull scale at construction, so a negative (inverted) strength has no effect on Grab specifically, unlike every displacement brush where sign flips the direction.
- Task 13: chose **mirror-the-stamp-and-requery** over a precomputed mirror-vertex mapping (the spec explicitly allows either). A vertex-index mapping would need to be derived either from generation-time knowledge (the surface-of-revolution's per-ring column index — fragile, since `SculptMesh` itself carries no such metadata and a future remesh's output has no reason to preserve exact index parity) or from a position-based nearest-match pass with its own tolerance/failure-mode questions. Mirroring the stamp and re-querying the existing spatial hash instead needs no new persistent structure, needs no rebuild step of its own on topology change (there is nothing to rebuild — it's computed fresh per stamp from whatever the live spatial hash already returns), and is brush-agnostic by construction: `symmetricStamps(stamp, axis)` just expands one `Stamp` into one or two before the existing query→kernel→normals pipeline runs, so it works identically for all six stamp brushes with zero brush-specific code.
- Task 13: verified the mirror-equal-displacement acceptance criterion two ways — a direct `symmetricStamps` unit test on a synthetic stamp, and an end-to-end test that runs a real Draw stroke (expanded via `symmetricStamps`) on a generated sphere and confirms the vertex nearest the mirrored position moved by the mirrored delta within float tolerance. The end-to-end test's mirror-vertex lookup is a brute-force nearest-position search written independently of `mirrorStamp`/`symmetricStamps`, so it's an honest check of the actual geometric outcome, not a tautology against the same negation logic being tested.
- Task 13: **Grab needed a separate `SymmetricGrabStroke` wrapper**, not `symmetricStamps` — Grab is stroke-stateful (Task 12) rather than a per-stamp kernel call, so there's no single `Stamp` to expand mid-drag. `SymmetricGrabStroke` instead constructs two independent `GrabStroke`s at touch-down (the primary at the real point, a second at the spatial-hash query result for the mirrored point) and, each update, drives the mirrored one from `mirrorHit(hit)`. The two instances share no state; a vertex set that happens to straddle both queries (near the seam) is simply translated independently by each, the Grab equivalent of `symmetricStamps`'s deliberate double-stamp-at-the-seam behavior.
- Task 13: FR-11 ("toggling symmetry off mid-session affects only subsequent strokes") is satisfied structurally rather than by any explicit toggle-tracking code — `axis` is an explicit argument to `symmetricStamps` and an explicit constructor argument to `SymmetricGrabStroke`, never live mutable module state, so a caller (the Task 15 facade) that captures the axis once at `beginStroke`/grab-construction and holds it for that stroke's duration automatically gets "changing the setting doesn't retroactively affect what's already in progress" for free. Tested by constructing two `SymmetricGrabStroke`s with different axes and confirming the earlier one's behavior is unaffected by the later one's construction.
- Task 14: `RemeshHistoryEntry` (full before/after `SculptMesh` snapshots) is defined as part of the `HistoryEntry` union now — matching the spec's Data Model — but `SculptHistory` has no `pushRemesh` yet; that's genuinely Task 16's job, once `remesh()` exists to produce an `afterMesh`. What Task 14 *does* build generically (so Task 16 doesn't need to touch it again) is the eviction algorithm's entry-kind awareness: `entryByteSize`/`indexOfEntryToEvict` already switch on `kind`, so a future remesh entry is correctly preferred for eviction over stroke entries without any change to `evictOverBudgetEntries`.
- Task 14: `undo()`/`redo()` return the popped `HistoryEntry` itself (not just a boolean), so a stroke entry's position restore happens inline here (self-contained, no topology change involved) while a remesh entry — once Task 16 adds one — can be handed back to the facade to apply its own mesh-swap + adjacency/spatial-hash rebuild, which this module has no business knowing how to do.
- Task 14: memory bound is a byte budget (`maxBytes`, constructor-configurable, defaulting to 256 MiB), not a fixed entry count — matches the spec's "history exceeds bound" framing (a byte budget) rather than a step-count cap. Eviction only ever removes entries from *outside* the protected most-recent `MIN_UNDO_STEPS` (50) window, so the ≥50-sequential-undo acceptance criterion holds regardless of how small the configured budget is; tested with a deliberately tiny budget (500 bytes) forcing eviction well before naturally filling 60 small entries, confirming exactly the newest 50 survive.
